## 二、教学模式层(Pedagogy Layer)——本次新增的核心设计

### 2.1 要解决的问题(创始人原话引发)

> "AI 不清楚每个学生的学习方式。我喜欢苏格拉底法,但 AI 总是把基本概念让我猜;还有的人喜欢 drill。"

这是通用 AI 当老师的一个典型病症:**把苏格拉底式当万能钥匙**。教学法上这是明确错误——引导式提问的前提是学生已有可供激活的知识;**首次接触的新概念必须直接讲清楚**(定义→物理直觉→最小例子),然后用提问检验理解,而不是让学生凭空猜定义。同时,不同学生有稳定的个人偏好:有人要对话引导,有人要高密度刷题。

结论:"怎么教"不能靠模型每次即兴发挥,要像"教什么"一样变成**可审核、可修订的数据资产**。

### 2.2 设计:教学策略卡 + 模式选择矩阵 + 偏好档案

**(1)教学策略卡(pedagogy cards)——"怎么教"做成数据,与知识卡片同构**

每张策略卡定义一种教学模式:适用前提、回合节拍、允许/禁止行为、话术样例、退出条件。初始四张:

| 卡 | 适用场景 | 节拍 | 硬性禁令 |
|---|---|---|---|
| `direct` 讲授模式 | **新知识首次学习** | 直接讲清(定义→直觉→最小例子)→ 检查性提问(检验理解)→ 学生复述要点 | **禁止对学生未学过的概念提问"你觉得是什么"** |
| `guided-repair` 引导修复 | 学过但薄弱 / 答错之后 | 1-2 个探测问题定位断点 → 从断点处补讲 → 回到原题重走 | 禁止从头全部重讲(浪费时间),禁止不探测直接讲 |
| `socratic` 苏格拉底阶梯 | **解题应用**(知识已具备) | 先问卡点 → 思路提示 → 方法提示 → 公式提示 → 解析 → 变式检验(即教学引擎的状态机) | 禁止跳级泄答案;学生坚持两次后给解析但必跟变式 |
| `drill` 刷题模式 | 已掌握内容的巩固提速 | 快节奏连续出题;对了只说"对,下一题";错了一句话点破错处;每 N 题小结 | **禁止每题长篇讲解**;连错两题自动切出 drill 转 guided-repair |

**(2)模式选择矩阵(状态机规则,不靠模型悟)**

| 内容状态(来自掌握度记录) | 默认模式 | 学生偏好可否覆盖 |
|---|---|---|
| 新知识(该考点无学习记录) | `direct` | 部分可:偏好"先试后讲"的学生可先给一道探路题,但**概念讲授本身不可被"猜谜"替代** |
| 薄弱(掌握度低 / 刚答错) | `guided-repair` | 可覆盖为 direct(学生要求"直接讲") |
| 应用(讲完做题 / 学生带题来) | `socratic` | 可覆盖为 direct(直接讲解题) |
| 巩固(掌握度高,提速期) | `drill` | 可覆盖为 socratic(想细抠的学生) |

**(3)学生教学偏好档案(style profile)**

- **初始化**:首次使用问 3 个问题——①新东西想先听讲解还是先试着做?②做错时想直接看讲解还是被引导找错?③练习想大量刷还是精讲一题?
- **显式调整(最重要的通道)**:学生随时一句话即可改档案——"别让我猜,直接讲"→ 意图识别 → 写入档案 → **立即且永久生效**,并明确确认:"好,以后新概念我直接讲清楚,做题时再用提问引导你——随时可以再改。"
- **隐式信号**:连续跳过提示直接要解析、对提问表现不耐烦 → **不自动改档案**,而是提议:"我注意到你更想直接看讲解,要不要把默认模式改成讲授型?"(自动改会显得自作主张,提议才像好老师。)

**(4)"AI 自主构建教学逻辑骨架"的落地方式**

策略卡初稿由 Claude 起草(教学法知识是模型强项),创始人实测修订;每张卡有版本号;实测中每个"不像好老师"的瞬间都定位到某张策略卡并修卡——与知识卡片同一套"起草→人审→实测→修卡"流程。**教学法由此成为与知识库并列的第二种可审计资产。**

### 2.3 对现有设计的影响

- 教学引擎(《研究资料/设计稿-教学引擎与超纲控制》)的状态机原设计只覆盖 `socratic`,现升级为:**状态机先按矩阵选策略卡,策略卡决定本轮注入的教学法指令**。
- 学生档案 schema 增加 `style_profile` 字段(三个偏好维度 + 修改历史)。
- 里程碑 M2 的验收标准增加:①新概念首次学习时 AI 直接讲授、不让学生猜;②"别让我猜"一句话能永久改写模式;③drill 模式下节拍明显快于对话模式。

## 三、题库接入方案(2026-07-02 创始人已确认)

**题库 = 讲义自带的约 310 道题**(不另外提供题库文件)。录入分两阶段:

### 3.1 阶段一:题目索引(现在做)

创始人指示:现在只需要知道每道题**考了什么**和**什么时候给学生**,答案与解析**由创始人后续提供**。

每道题建立索引记录:考点标签(映射到已定稿知识树的 41 个编号,结构化输出锁死编号集)、题型、模型/情境标签、难度建议(D1-D5)、**出题时机(stage)**:

| stage | 含义 | 对应教学环节 |
|---|---|---|
| `concept_check` | 讲完概念立即检验理解(判一判/想一想/直白辨析题) | 讲授模式(direct)的检查环节 |
| `basic_practice` | 单知识点直用,合格考水平 | drill 模式弹药、讲后首练 |
| `standard` | 常规综合,等级考中档 | 苏格拉底讲题、每日练习主体 |
| `advanced` | 多过程/临界/字母运算压轴 | 冲刺练习、能力上探 |
| `review_test` | 章末水平测评题 | 整卷检测模式 |

索引成果:`content/questions/index-x3.json`(选题引擎直接消费)+《04-题目索引-选必三》(人读版)。

### 3.2 阶段二:答案与解析(等创始人提供)

创始人发答案/解析后逐题挂接:标答 + 分步评分点(AI 转写、创始人快审)→ 该题解锁"批改"能力。索引里 `answer_status` 字段跟踪:`pending`(仅可呈现与讲评,批改降级为"参考模式")→ `ready`(可正式批改)。**没有标答的题不进入正式批改管线**(决策 #7 不放松)。

### 3.3 选题引擎

规则引擎按(考点 × stage × 难度 × 学生掌握度 × 复练队列到期项)从索引检索,**AI 不参与"出哪道题"的决策**——可解释、可复盘、不抽风。AI 的角色只在:录入时打标签、讲评时讲题。
