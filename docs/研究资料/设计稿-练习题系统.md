# 练习题系统设计方案
## ——高考物理 AI 私教:从题目来源到屏幕呈现到批改反馈

> 文档定位:本方案是练习题子系统的完整设计,读者为工程实现者(Claude Code)和创始人(产品/教研/测试)。每个关键决策附「为什么」的通俗解释;凡 AI 当前做不到"完美"之处,均如实标注真实水平、失败模式与兜底。全文按「骨架版(V0,第一版必须有)/ 完整版(V1+,逐层叠加)」双档给出。

---

## 0. 总览:一条数据流,五个设计原则

练习题系统的完整数据流:

```
题目入库(来源→结构化→审核)
   → 出题(难度/考点选题引擎)
   → 呈现(排版 + 公式渲染 + 示意图渲染)
   → 作答(选择/填空/分步计算 + 阶梯提示)
   → 批改(客观题本地判分 / 主观题 AI 过程批改)
   → 回写(错题本 + 考点掌握度 + 复练队列)
   → 影响下一次出题(闭环)
```

五个贯穿全文的设计原则(每个都来自调研中的真实教训):

1. **能用代码判的绝不用 AI 判。** 选择题、数值填空用程序判分,100% 确定;AI 只批改程序判不了的过程题。(教训来源:学而思 AI 批改"把对判成错"被家长公开投诉;Khanmigo 早期连 343-17 都算错。)
2. **AI 批改前必须喂标准答案。** 让 AI"裸批"是幻觉重灾区;批改 prompt 中注入人工审核过的标准解答与评分点后,可靠性发生质变。(Khan Academy 官方博客明确验证过此做法,是可直接复用的工程结论。)
3. **AI 不直接画图,只填参数。** 示意图由 LLM 输出结构化 JSON、确定性渲染器绘制;一步生成原始 SVG 不可靠。(Hacker News 上的物理私教原型、DiagramEval 等研究均指向同一结论。)
4. **数值必须程序验算。** 无论是批改还是出题,涉及数值的一律由代码重算,不信任模型心算。
5. **每道进入学生视野的题,必须有人(创始人)看过。** 骨架版题量小,全量人审完全可行;这是产品可信度的底线。

## 1. 网页呈现技术

### 1.1 公式渲染:KaTeX 方案

**决策:全站默认 KaTeX,不引 MathJax。**

给创始人的解释:网页本身不会显示 `½mv²` 这种数学排版,需要一个"公式排版库"把 LaTeX 代码(一种描述公式的文本语法)变成漂亮的公式。业界只有两个选择,KaTeX 快、轻,MathJax 功能全但重。高中物理公式的复杂度(矢量箭头、分式、根号、上下标、希腊字母)在 KaTeX 能力范围内绰绰有余,选快的那个。KaTeX 本身就是可汗学院出品并在其生产环境使用,成熟可靠。

工程要点:

- **存储约定:** 题干、选项、解析统一为 **Markdown + LaTeX 混排**,行内公式用 `$...$`,独立公式用 `$$...$$`。数据库里存源文本,前端渲染。这是所有 AI 模型最熟悉的格式,Claude 生成题目时零转换成本。
- **渲染方式:** 前端用 `katex` + `auto-render` 扩展遍历题目容器;开 `throwOnError: false`,渲染失败时显示原始 LaTeX 源码(红色)而不是白屏,同时上报日志。渲染失败的题会被日志捕获,进创始人审核队列修文本。
- **书写规范(写入出题/录题的系统提示词,保证全库一致):**
  - 矢量:`$\vec{F}$`;单位:`$v = 5\,\text{m/s}$`(单位用正体,`\text{}` 包裹);
  - 科学计数法:`$3.0\times10^{8}\,\text{m/s}$`;
  - 禁止用 Unicode 上标字符(如 `m²`)混排,统一走 LaTeX,避免风格花掉;
  - 中文与公式之间由渲染样式控制间距,源文本不手动加空格。
- **样式:** 公式字号与正文一致(继承 1em),行内公式设 `vertical-align` 微调与中文基线对齐;独立公式居中、上下留 0.75em。

骨架版 = 完整版,此项一步到位,没有分档必要。唯一的完整版补充:若未来遇到 KaTeX 不支持的命令(概率很低),对单题降级 MathJax,不全局引入。

### 1.2 物理示意图:AI 生成的真实水平、参数化图形库、兜底链

这是整个呈现层**技术风险最高**的部分,必须先讲实话。

#### 1.2.1 AI 直接生成 SVG 的真实质量(诚实评估)

基于调研(DiagramEval、FeynmanBench、实际原型实践)和对当前模型能力的合理判断:

| 图形类型 | AI 一步生成原始 SVG 的可用率(估) | 典型失败模式 |
|---|---|---|
| v-t / x-t / F-x 等函数图像 | 高,但**不该用 AI 画**——应由代码按函数式绘制 | 刻度不均匀、折线拐点位置与题意数值不符 |
| 单物块+受力箭头(水平面) | 中等偏上,约 60-80% 可看 | 箭头长度比例失真、标注文字压线重叠 |
| 斜面、传送带、板块 | 中等,约 50-70% | 斜面角度与标注不符、物块悬空或嵌进斜面、摩擦力方向画反 |
| 电路图 | **低,不可接受** | 元件符号不符合国标(GB/T 4728)、导线交叉混乱、节点连接错误——学生会被错图直接教坏 |
| 光路图(折射/透镜成像) | **低,不可接受** | 光路角度必须几何精确(折射角、焦点位置),AI 画的角度是"大概齐",物理上是错的 |
| 复杂组合(圆轨道+平抛、多滑轮) | 低 | 几何关系崩坏 |

**结论:必须做预制参数化图形库,这不是"要不要"的问题,是电路图和光路图根本没有别的可行方案。** 靠谱的架构是调研中验证过的三层:**LLM 输出场景 JSON → 确定性渲染器画 SVG → 渲染不了的降级**。

给创始人的解释:好比宜家家具——AI 不徒手做木工(容易歪),而是从我们预制的"零件库"(斜面、物块、箭头、电阻符号)里挑零件、填尺寸,由程序精确组装。程序组装的图,角度、比例、符号永远是对的。

#### 1.2.2 参数化图形库(Figure DSL)设计

每道题的图存为一个 JSON 场景描述,渲染器是一组纯 TypeScript 函数(输入 JSON → 输出 SVG 字符串),无外部依赖、可服务端预渲染。

**场景类型清单与分期:**

| 场景 `scene` | 覆盖的高频题型 | 分期 |
|---|---|---|
| `graph`(通用函数图:v-t、x-t、F-t、U-I…) | 运动学图像题、图像信息题(高考每卷必有) | 骨架版 |
| `flat_block`(水平面物块+力箭头) | 牛顿定律、摩擦力 | 骨架版 |
| `incline`(斜面+物块+力箭头) | 斜面模型全家桶 | 骨架版 |
| `conveyor`(传送带) | 传送带模型 | 骨架版 |
| `blocks_stacked`(板块模型) | 板块相对滑动 | 骨架版 |
| `circuit_basic`(电源/电阻/开关/电表/滑动变阻器/电容,网格化布线) | 闭合电路、电学实验 | 骨架版(工作量最大的一个,但不可省) |
| `vertical_circle`(竖直圆轨道+平台) | 圆周运动压轴 | 完整版第一批 |
| `projectile`(平抛/类平抛轨迹) | 抛体运动 | 完整版第一批 |
| `spring`、`pulley`(弹簧、滑轮组) | 弹簧模型、连接体 | 完整版第一批 |
| `em_field`(匀强场区域+带电粒子轨迹) | 电磁场偏转、复合场 | 完整版第二批 |
| `optics`(界面折射/棱镜/双缝) | 光学 | 完整版第二批 |
| `rail`(导轨切割) | 电磁感应压轴 | 完整版第二批 |

**场景 JSON 示例(斜面):**

```json
{
  "kind": "param",
  "scene": "incline",
  "params": {
    "angle_deg": 37,
    "incline_direction": "right_up",
    "block": { "label": "A", "width_ratio": 0.18 },
    "forces": [
      { "type": "gravity", "label": "mg" },
      { "type": "normal", "label": "N" },
      { "type": "friction", "dir": "up_slope", "label": "f" },
      { "type": "applied", "angle_deg_from_slope": 0, "dir": "up_slope", "label": "F" }
    ],
    "annotations": [ { "type": "angle_arc", "label": "37°" } ]
  },
  "alt_text": "物块A置于倾角37度的斜面上,受重力mg、支持力N、沿斜面向上的摩擦力f和沿斜面向上的拉力F"
}
```

**电路场景 JSON 示例:**

```json
{
  "kind": "param",
  "scene": "circuit_basic",
  "params": {
    "grid": [4, 3],
    "components": [
      { "id": "E", "type": "battery", "cell": [0, 0], "label": "E, r" },
      { "id": "S", "type": "switch", "cell": [1, 0], "label": "S" },
      { "id": "R1", "type": "resistor", "cell": [2, 0], "label": "R₁" },
      { "id": "R2", "type": "rheostat", "cell": [2, 2], "label": "R₂" },
      { "id": "A1", "type": "ammeter", "cell": [3, 1], "label": "A" }
    ],
    "wires": [["E","S"],["S","R1"],["R1","A1"],["A1","R2"],["R2","E"]]
  },
  "alt_text": "电源E(内阻r)、开关S、定值电阻R1串联,再与滑动变阻器R2、电流表A构成回路"
}
```

渲染器职责:元件符号按国标绘制(工程侧照着 GB/T 4728 常用符号画一遍,一次性成本);网格自动布线走直角;标签自动避让(简单规则:优先放元件上方,冲突则下方)。

**每个场景渲染器附一个"参数校验器"**(如 `angle_deg` 必须在 10-75 之间、`wires` 必须构成连通回路),AI 填错参数时校验失败即触发降级,而不是画出鬼图。

#### 1.2.3 兜底降级链(每道题的图按此顺序取用)

```
1. 人工审核过的场景 JSON(入库题标配)
2. 人工上传的图片(真题原图截图——内测阶段合法可用)
3. AI 即时生成场景 JSON + 校验通过 → 渲染(用于即兴变式题)
4. 校验失败 → AI 直接生成 SVG,页面标注"示意图由AI生成,以文字描述为准"
   (仅允许用于 graph 和 flat_block 两类低风险图;电路/光路禁止走这层)
5. 无图,题干补充完整文字描述(alt_text 升级为正文)
```

**骨架版对图的务实要求:** 内测阶段所有入库题的图,由创始人在录题时二选一——用参数化库配出来(录题工具里提供表单+实时预览),或直接贴真题截图。**不依赖 AI 即时画图。** AI 即时画图只服务于"会话内即兴变式题",且限定低风险场景。

#### 1.2.4 图的呈现规范

- SVG 内嵌页面(非 img 标签),继承主题色,支持暗色模式变量;
- 移动端宽度 100%、最大高度 45vh;
- 所有图必带 `alt_text`(录题时 AI 起草、人工确认)——既是无障碍属性,更是第 5 层降级的正文来源,还是喂给批改模型的图形文字化描述(批改时模型看文字比看图可靠)。

### 1.3 题目排版样式规范

题目卡片自上而下的固定结构:

```
[元信息行] 题型徽章 · 来源标签(如"2024·新课标I卷") · 难度(●●●○○) · 考点标签(可点击)
[题干区]   Markdown+KaTeX 渲染;数据行(已知量)自动加浅色底纹强调
[图区]     居中,下方图注(可选)
[作答区]   随题型变化(见第 2 章)
[操作行]   提示按钮(阶梯) · 标记本题 · 提交
```

样式硬指标(移动端优先——学生主力设备是手机):

- 正文 17px/行高 1.8,题干与选项之间 16px 间距;选项行高不小于 44px(拇指可点);
- 数值与单位永不拆行(`5\,\text{m/s}` 渲染为不可断行单元);
- 不定项选择题在题型徽章处显式标注"不定项",并在选项区顶部提示判分规则(见 3.1)——高考物理多选题的"漏选得部分分"规则学生必须清楚;
- 一屏一题(练习会话模式),不做长列表滚动刷题——本产品是私教不是题海,呈现形态要强化"做一题、吃透一题"。

## 2. 作答交互设计

### 2.1 三种题型的作答方式

**单选/不定项选择:** 点选选项,不定项可多选;选中态明显;提交前可改。骨架版即终态。

**填空题:** 按空的答案类型给不同控件:

- `numeric`:数值输入框 + 单位选择(下拉,预置本题合理单位集,默认标准单位)。数值与单位分离是刻意设计——学生只管填数,单位错配由程序换算判断,消灭"答案对但单位写法不同被判错"这种最伤信任的误判。
- `expression`(表达式空,如"用 m、g、θ 表示"):文本输入 + 实时 KaTeX 预览(见 2.4)。
- `text`(概念填空,如"增大/减小"):短文本或预置选项二选一,能预置就预置。

**计算大题:** 核心是分步作答,见 2.2。

### 2.2 计算题的"分步作答"设计

**为什么分步:** 一次性收一大段解答,AI 批改要同时干"切分步骤+逐步判断+定位首错"三件事,错误率叠加;把结构化的活前置到作答交互里,批改难度直线下降。同时,分步本身就是教学法——强迫学生按"受力分析→列方程→解算"的规范路径思考,这正是线下好老师逼学生做的事。

**数据结构:题目定义 `steps[]`,每步一个作答单元:**

```json
{
  "steps": [
    {
      "id": "s1",
      "ask": "对物块进行受力分析:它受到哪几个力?",
      "type": "force_check",
      "choices": ["重力", "支持力", "沿斜面向上的摩擦力", "沿斜面向下的摩擦力", "拉力F", "离心力"],
      "reference": ["重力", "支持力", "沿斜面向上的摩擦力", "拉力F"],
      "rubric": "选'离心力'记概念错误(cpt_fictitious_force);漏摩擦力记漏条件"
    },
    {
      "id": "s2",
      "ask": "沿斜面方向列出牛顿第二定律方程",
      "type": "equation",
      "input": "text_with_preview",
      "reference": "F - mg\\sin\\theta - \\mu mg\\cos\\theta = ma",
      "reference_equivalents": ["F - mg sinθ - μmg cosθ = ma", "ma = F - mg sinθ - f, f = μN, N = mg cosθ"],
      "rubric": "正弦余弦搞反 → 建模错(mdl_incline_decompose);漏摩擦项 → 漏条件"
    },
    {
      "id": "s3",
      "ask": "求物块的加速度大小",
      "type": "numeric",
      "reference": { "value": 2.0, "unit": "m/s^2", "tolerance_rel": 0.02 }
    }
  ]
}
```

交互形态:步骤卡逐张展开(手风琴),完成上一步才解锁下一步;也允许"跳过此步"(记为空白,批改时如实反映)。每步独立可提示、独立被批改。

**分档:**
- **骨架版:** `force_check`(勾选题,程序可判)+ `equation`(文本+预览,AI 批)+ `numeric`(程序判)三种步骤类型。这已覆盖 80% 计算题的骨架。
- **完整版:** 增加"在图上拖拽绘制力箭头"的交互式受力图作答(开发成本高、教学价值大,放 V1)、方程模板填空(给出 `__ - __ = ma` 挖空)、多问大题的问间依赖(第2问用第1问结果,批改时容错"连带分"——前步错导致后步数值错但方法对,只扣前步的分,这是高考评分的真实规则,必须实现)。连带分逻辑骨架版先在批改 prompt 里用自然语言要求,完整版做成结构化规则。

### 2.3 阶梯提示(Hint Ladder)

固定三级,每题录入时随题定义(AI 起草、人工审核):

| 级别 | 内容 | 示例(斜面题) | 对掌握度的影响 |
|---|---|---|---|
| H1 思路提示 | 只指方向,不含公式 | "先想清楚:物块沿斜面方向受哪几个力?它们的合力决定加速度。" | 得分打 9 折 |
| H2 方法提示 | 点名定理/公式,不代入 | "沿斜面方向用牛顿第二定律,注意重力要分解:沿斜面分量是 mg sinθ。" | 得分打 6 折 |
| H3 完整解析 | 全解 | 标准解答全文 | 本题记 0 分,且**自动标记"未独立完成",3 天后一道变式题进入复练队列** |

交互:作答区旁常驻一个"卡住了?"按钮,点开后逐级展开,已解锁级别保持可见;级别只能顺序解锁。系统记录 `hint_level_used` 随提交一起入库。

**为什么看完解析要压变式重练:** 防"点开解析→抄→假掌握"刷穿系统。看解析不可耻(私教本来就该讲),但看过解析的题不能算你会,必须换个数字换个皮再考你一次——这是本产品和普通题库的核心差别之一。

骨架版 = 上述全部(提示文本在录题时生成审核)。完整版:H1/H2 由 AI 根据学生已答内容动态生成(针对"你卡在哪"而不是通用提示)——更像真私教,但依赖批改中间态,放 V1。

### 2.4 公式输入的现实困难与解决方案(诚实评估)

**现实:高三学生 100% 不会 LaTeX,也不该学。** 三层方案:

**方案 A(骨架版主力):简化文本约定 + 实时预览 + 符号快捷条。**

- 约定:`^` 上标、`_` 下标、`/` 分式、`sqrt()` 根号、`*` 可省略;希腊字母和特殊符号从输入框上方的快捷条点选插入(一排按钮:`θ μ ω ρ α λ π ² ½ √ × ± °`,按物理高频排序);
- 输入框下方**实时渲染预览**:学生打 `v0^2 = 2*g*h`,下方即刻显示 v₀² = 2gh 的排版效果。预览是关键——学生看见"机器理解成了什么",输入歧义当场消除;
- 解析器:写一个宽容的 tokenizer 把简化语法转 LaTeX(约 200 行代码),解析不了的原样透传给批改模型(Claude 读 `mgsinθ` 这种松散写法的能力很强,不必强求前端解析完美)。

**方案 B(完整版第一批):拍照上传手写解答。**

- 调研数据:多模态大模型对真实手写数学作业的可接受转写率约 88%(GPT-4.1-mini 配专用提示词;Claude 同量级)。**88% 意味着约每 8 题有 1 题转写有瑕疵**,直接拿去批改会把 OCR 错误和解题错误搅在一起,学生被误判后无从申辩。
- 因此交互上强制**"转写确认"**环节:上传照片 → Claude 转写为结构化文本+公式 → 回显给学生"我认出来的是这些,对吗?"→ 学生修正确认 → 才进批改。把识别错误在批改前拦掉,这一个 UX 决策能消掉一半的误判投诉。
- 骨架版不做拍照的原因:内测用户是创始人自己,打字无障碍;拍照管线(上传、压缩、转写、确认 UI)是完整版接真实学生前的第一优先级。

**方案 C(明确不做):** 可视化公式编辑器(MathQuill 类点击搭公式)。对高中生学习成本高、移动端体验差、开发投入大,性价比低于 A+B。

## 3. 批改逻辑

### 3.1 客观题:程序判分,不走 AI

- **单选:** 精确匹配。
- **不定项:** 按高考规则判:全对满分;所选均正确但不全 → 部分分(通常满分的一半,题目可配置);含错选 → 0 分。规则写死在代码里,题目 JSON 可覆写。
- **数值填空:** 判分函数 `judge_numeric(student, reference)`:
  1. 单位归一(学生填 18 km/h、答案 5 m/s → 换算后比对;维护一张 SI 换算表);
  2. 相对误差容差(默认 2%,题目可配置——g 取 9.8 还是 10 会导致合理差异,录题时按题设标定);
  3. 有效数字宽容(4 与 4.0 与 4.00 同判)。
- **表达式填空(诚实说明:这是客观题里唯一不 100% 可靠的):** 符号表达式等价判断(`mg sinθ·μ` vs `μmgsinθ` vs 化简形式)程序上是 CAS 难题。骨架版方案:标准答案存 3-5 个等价形式白名单 → 归一化字符串比对(去空格、排序乘积项)→ 比对失败的**不判错**,转 AI 判等价(prompt:只回答 equivalent/not/unsure),unsure 一律按"待复核"温和呈现。宁可少判对错,不可错判。

给创始人的解释:客观题判分是纯代码,这部分对错判断和计算器一样可靠,永远不会"抽风"。产品里 60-70% 的判分量走这条路,这是可靠性的基本盘。

### 3.2 计算题过程批改:管线、错因体系、输出结构

#### 批改管线(骨架版即此形态)

```
输入打包:
  题目全文(含图的 alt_text 文字描述)
  + 标准解答 solution(人工审核过)
  + 分步评分点 rubric(每步的 reference + 常见错误→错因码映射)
  + 学生逐步作答(含使用的提示级别)
      ↓
Claude 批改调用(强制 JSON 输出,temperature 低)
      ↓
数值复核:凡 verdict 依赖数值比较的,由后端代码重算一遍
  (模型只判"方法/概念",数字对不对由程序说了算 —— Khanmigo 计算器教训)
      ↓
批改结果 JSON → 校验 schema → 呈现给学生 + 回写数据层
```

#### 错因分类体系(骨架版:4 大类 + 约 20 子码,创始人审定)

讯飞做到 4000+ 标签用了多年和专家团队;我们骨架版做**4 大类、20 个左右子码**,粒度足够驱动"归因→补课"闭环,且 AI 归类准确率随类目减少而上升(类目越细越容易归错)。

| 大类 | 码 | 子码示例 |
|---|---|---|
| 概念错误 | `cpt` | `cpt_force_pair`(作用力反作用力混淆)、`cpt_fictitious_force`(离心力当真实力)、`cpt_energy_sign`(功的正负) |
| 建模错误 | `mdl` | `mdl_incline_decompose`(斜面分解 sin/cos 反)、`mdl_process_split`(多过程未分段)、`mdl_object_choice`(研究对象选错)、`mdl_critical_state`(临界条件没找到) |
| 计算错误 | `cal` | `cal_arithmetic`(算术错)、`cal_algebra`(移项/代数错)、`cal_unit`(单位换算错) |
| 审题遗漏 | `omt` | `omt_condition`(漏用已知条件)、`omt_question`(答非所问)、`omt_implicit`(隐含条件没挖出,如"恰好"="临界") |

子码表由创始人一次性审定(半天工作量),之后每季度按真实批改数据增补。**这套错因体系 + 你写的解析,就是调研摘要五里说的"自有版权资产"。**

#### 批改输出 JSON(schema 固定,供工程直接实现)

```json
{
  "submission_id": "sub_20260702_001",
  "overall": { "score": 6, "full_score": 10, "verdict": "partially_correct" },
  "steps": [
    { "step_id": "s1", "verdict": "correct", "comment": "受力分析完整正确" },
    { "step_id": "s2", "verdict": "wrong",
      "error_type": "mdl_incline_decompose",
      "evidence": "学生方程为 F - mg\\cosθ - ... ,重力沿斜面分量应为 mg\\sinθ",
      "comment": "重力分解时把 sinθ 和 cosθ 用反了。记住:沿斜面方向永远配 sinθ(θ 是斜面倾角)。" },
    { "step_id": "s3", "verdict": "consequential_error",
      "comment": "数值错误由第2步方程错误导致,解方程过程本身正确,按连带处理不重复扣分" }
  ],
  "first_error_step": "s2",
  "primary_error_type": "mdl_incline_decompose",
  "confidence": 0.88,
  "nonstandard_solution_detected": false,
  "feedback_text": "前面的受力分析很扎实。问题出在重力分解……(面向学生的鼓励式总评,2-4句)"
}
```

呈现规则:逐步显示对错标记与批注;`confidence < 0.7` 时,措辞从"这一步错了"降级为"这一步可能有问题,我的判断是……如果你认为自己是对的,点这里让我重新看"。

### 3.3 回写:错题本与掌握度

**批改完成后的三个写入动作:**

1. **作答记录 `attempts` 表:** `{question_id, steps_result, score_ratio, hint_level_used, error_types[], duration_sec, timestamp}` —— 全量留存,是一切分析的原料,也是批改质量回归测试的重放数据。
2. **掌握度更新(按考点):** 每个考点维护 `mastery ∈ [0,1]`,指数移动平均:
   `mastery_new = mastery + α × (effective_score − mastery)`
   其中 `effective_score = score_ratio × hint_discount(0.9/0.6/0)`,`α` 基础值 0.3,题目难度高于学生当前水平时上调(高难度题的信号更有价值)。一题多考点时,`primary_error_type` 对应的考点吃全额更新,其余考点吃半额。
   给创始人的解释:掌握度就是"最近表现的加权平均,越近的表现权重越大"——骨架版刻意用这个小学生都能对账的公式,不上黑盒算法,方便你自测时验证"系统对我的判断准不准"。
3. **错题本写入:** 凡 `score_ratio < 0.8` 或用过 H3 的题,生成/更新错题条目:

```json
{
  "wrongbook_id": "wb_0142",
  "question_id": "q_0033",
  "knowledge_points": ["kp_newton2_incline"],
  "error_types": ["mdl_incline_decompose"],
  "status": "pending_review",
  "review_stage": 0,
  "next_review_at": "2026-07-03",
  "history": [{ "date": "2026-07-02", "score_ratio": 0.6, "hint": 1 }]
}
```

### 3.4 AI 批改的真实可靠性与兜底(必须诚实的一节)

**真实水平预估(在"有人工标准答案+rubric+分步作答"的最优条件下):**

- 步骤对错判定与人类一致性:预计 **85-93%**。讯飞公布的 97% 是全力投入多年、海量标注数据的结果,我们初期达不到,不承诺;
- 错因归类(4 大类)准确率:预计 80-90%;子码级会更低(70-85%);
- **最危险的失败模式按伤害排序:** ① 把对判错(学生自信被打击、对产品信任崩塌——学而思被投诉的正是这个);② 对非常规但正确的解法误判(学生用动量做、标准答案用能量做);③ 错因归类跑偏导致后续补课开错药方;④ 批注里自己算错数(用数值复核层压制)。

**四道兜底(骨架版全部要有,这是产品底线不是锦上添花):**

1. **非常规解法检测:** prompt 要求模型显式输出 `nonstandard_solution_detected`;为真且最终数值与标准答案一致时,呈现改为"你的解法和参考解法不同,结果正确。过程判定仅供参考",且**该次不更新掌握度、不进错题本**,转人工复核队列。
2. **"我认为批错了"按钮:** 每个批改结果旁常驻。点击 → 该题判定冻结(不影响掌握度)→ 进创始人复核队列 → 复核结论回写并(完整版)向学生推送更正。内测期这个按钮同时是创始人给批改质量打标的工具。
3. **置信度分级措辞:** 见 3.2,低置信不说死话。
4. **批改重放回归:** 每次改批改 prompt,对留存的历史 submission 全量重跑、对比人工标定结论,防止"改好一处、坏掉三处"。工程上就是一个脚本 + 一份创始人标定的金标集(golden set,首批 50 题人工标定,创始人约 1 天工作量)。

**完整版叠加:** 双模型交叉批改(两次独立批改结果不一致 → 自动转人工)、按错因子码的准确率看板、学生申诉率作为核心质量指标(目标 < 3%)。

## 4. 题目来源策略

### 4.1 版权现实(以调研摘要为准)

- 高考真题受著作权保护,权利人是命题机构;但**内部自测阶段使用真题风险极低**;
- **解析、错因标注、考点标签体系是自建版权资产**(菁优诉豆丁案确认),从第一题开始就要把"题目本体"和"我们的解析层"在数据结构上分开存(`source.kind` 字段 + 解析独立字段),将来商用时资产边界清晰;
- 商用阶段的安全区:自编题、改编题(改数字改情境)、自有解析层;明确侵权区:批量复制真题+他人解析并收费;
- 落库动作:每题必填 `source` 字段:`{kind: "past_exam" | "adapted" | "original" | "textbook", origin: "2024 新课标I卷 T14", province_tags: [...], commercial_ok: false}`。`commercial_ok` 现在看着多余,商用前夜它就是"哪些题要下架/替换"的一键筛选器。

### 4.2 AI 改编真题生成变式题

**做法(这是性价比最高的扩题方式,也是"看完解析后重练"的弹药库):**

改编维度按风险从低到高:
1. **数字重抽样**(最安全):同一题面,数值在合理区间重抽,答案随之重算;
2. **情境换皮:** 斜面→滑梯,小车→冰壶,物理结构不变;
3. **求解目标互换:** 原题给 μ 求 a,变式给 a 求 μ;
4. **增减一问 / 增减一个过程**(风险最高,可能改变难度档,骨架版不做)。

**质量控制管线(每道变式题必过):**

```
原题 + 改编指令 → Claude 生成变式题 + 完整解答
   → [自动] 程序验算:让模型同时输出"验算脚本"(纯计算的 JS 函数,
     代入自己给的数值,检查每个方程闭合、最终答案复现),后端沙箱执行
   → [自动] 物理合理性规则检查(见 4.3 的 sanity 规则表)
   → [自动] 独立求解交叉验证:另开一次无解答上下文的调用,只给题面,
     独立解题;两次答案不一致 → 直接废弃
   → [人工] 创始人终审(数字重抽样类:抽检;情境换皮/目标互换类:全审)
   → 入库,source.kind = "adapted",parent_question_id 指向原题
```

**真实通过率预期(诚实):** 数字重抽样一次通过率约 85-95%(失败主要是抽出的数值导致物理不合理,如 μ>1);情境换皮约 70-85%(失败主要是新情境引入了原题没有的物理因素,如换成"雨中"就该考虑阻力);自动管线能拦住大部分,但**换皮类必须人审**——AI 判断"情境是否符合物理直觉"的能力就是它自己的短板,不能既当运动员又当裁判。

### 4.3 AI 原创题的物理合理性自检

**先说结论:骨架版不做 AI 原创题入库。** 原创题的失败模式比改编题多一个量级:数值不自洽(解出时间为负)、条件冗余或矛盾(给了三个条件只需两个且互相打架)、答案不唯一(条件不足,"合理即可"在物理计算题里是事故)、情境反直觉(自行车骑出 30 m/s)、以及最隐蔽的——**题目没错但不像高考题**(考查点松散、设问没有区分度),这最后一条只有教研(创始人)能判。

完整版引入原创题时的自检管线(在 4.2 管线基础上强化):

**自动 sanity 规则表(代码实现,随验算脚本执行):**

| 检查项 | 规则示例 |
|---|---|
| 量纲一致 | 验算脚本中所有方程两侧量纲标注一致 |
| 数值域 | 0 < μ ≤ 1.5;v < 340 m/s(常规情境);a 常规 < 50 m/s²;t > 0;质量、距离 > 0 |
| 解的唯一性 | 验算脚本对方程组求解,确认自由度为 0;二次方程双根时必须有明确舍根依据写进解析 |
| 答案"好看"度 | 高考风格偏好整数/一位小数结果;答案为 3.7182… 的题打回重抽数值(不是错,是不像高考题) |
| 条件利用率 | 解答中每个已知量至少被引用一次,否则标记"条件冗余" |

**加上双独立求解一致 + 人工终审,原创题的入库通过率预期 60-80%,人审环节不可去除。** 这意味着原创题的真实成本 ≈ AI 生成成本 + 每题 3-5 分钟人审,仍远低于人写题,但不是"无限免费出题机"。

### 4.4 骨架版起步方案(最务实路径)

**结论:骨架版 = 创始人精选真题/经典题 150-300 道 + AI 辅助结构化录入 + 全量人审。**

- 范围:配合"先骨架后叠层"策略,首批只覆盖 2-3 个章节(建议:匀变速直线运动 + 牛顿运动定律 + 图像题——摸底诊断和前几周教学用得上的),每个考点 8-15 题、铺满 5 档难度;
- 录入工具(第一批要做的内部工具,优先级不低于学生端):粘贴题目文本/上传截图 → Claude 转结构化 JSON(题干 LaTeX 化、拆步骤、起草解析/评分点/三级提示/错因映射/考点标签/难度建议/图形场景 JSON)→ 创始人在审核界面逐字段确认修改 → 入库;
- 为什么不从 AI 生成题起步:骨架版的目的是验证"呈现→作答→批改→归因"闭环,题目质量必须是无争议的常量,不能让"题本身有没有问题"成为混淆变量。真题是免费的质量金标准,内测使用无版权风险;
- 升级路径:V0 真题为主 → V1 加数字重抽样变式(服务复练队列)→ V1.5 加情境换皮 → V2 引入原创题(此时已有真实作答数据反哺出题质量)。

## 5. 难度分级与出题策略

### 5.1 难度体系:五档,对齐高考实际

| 档 | 名称 | 能力要求(对齐评价体系"四翼"与学业质量) | 高考对应 |
|---|---|---|---|
| D1 | 概念直用 | 单一知识点、公式直接代入;考"基础性" | 选择题前2-3题水平 |
| D2 | 基础综合 | 单一模型、两步推理 | 基础选择/填空,合格考上限附近 |
| D3 | 常规综合 | 2-3 个知识点、标准模型(斜面/传送带/常规电路),考"综合性" | 中段大题、实验题主体 |
| D4 | 建模迁移 | 新情境需自行建模、多过程分段、定性半定量分析,考"应用性" | 后段大题、多选压轴 |
| D5 | 压轴 | 多对象多过程组合、临界与分类讨论、字母运算,考"创新性" | 最后一题后两问 |

分布对齐:高考卷大致呈 基础:中档:难 ≈ 3:5:2(通行说法,非官方数据),即 D1-D2 约 30%、D3 约 50%、D4-D5 约 20%。**产品含义:一个目标 70 分的学生,练习配比应重仓 D2-D3,D5 少量见识即可——"按目标分数配难度"是私教区别于题海的核心逻辑,这个配比策略由创始人以配置表形式给出。**

定标方式:骨架版由 AI 建议 + 创始人手标(教研经验是唯一可用信号);完整版用真实作答数据校准——某 D3 题全体正确率持续 > 85% 则降 D2(简化 Elo:题目和学生互相定位),数据量到几千次作答后启用。

### 5.2 会话内动态调整:透明规则引擎,不用 AI 黑盒

**为什么用规则不用 AI 现场决定:** 出题策略必须稳定、可解释、可被创始人直接调参("连对两题就升档"改成"三题"是改一个数字);AI 现场决定会飘,且无法复盘"为什么给他出了这道题"。AI 的角色在批改和讲解,不在选题仲裁。

**会话结构(一次练习 = 6-8 题,约 25-40 分钟):**

```
热身 1 题:目标考点,难度 = 学生该考点当前档 − 1(建立进入感)
主体 4-6 题:按下方规则动态走
收尾 1 题:回顾题,来自错题复练队列(若有到期条目,优先塞这里)
```

**主体段调整规则(伪代码级,工程直接照写):**

```
初始档 = mastery_to_level(该考点掌握度)   // 0.0-0.3→D1, 0.3-0.5→D2, 0.5-0.7→D3, 0.7-0.85→D4, >0.85→D5
连续 2 题 effective_score ≥ 0.9 且未用提示  → 升 1 档(封顶:目标配比允许的最高档)
本题 0.5 ≤ score < 0.9 或用了 H1/H2       → 保持当前档,下一题换同考点不同模型
本题 score < 0.5 或用了 H3                → 同档出一道该题的变式(数字重抽样);
                                             变式再错 → 降 1 档 + 插入一段 AI 微讲解(90 秒,针对 primary_error_type)
                                             再错 → 结束该考点练习,标记"需要回炉讲课",移交教学模块
```

最后一条是私教气质的关键:**连错三次还继续喂题是题海行为;私教的正确动作是停下来重讲。** 练习系统要有"我教不动了,退回讲课环节"的出口。

### 5.3 错题本形态

**两个视图 + 一条队列:**

**视图一:按考点聚类(诊断视图)。** 考点卡片:错题数、掌握度、主错因分布条(如"斜面模型:60% 建模错,30% 计算错")。这个视图回答"我到底弱在哪",也是调整复习计划模块的数据入口。

**视图二:错题时间线(浏览视图)。** 按时间倒序,每条含题目缩略、当时的错误步骤和批注、错因标签。支持按错因筛选("把我所有'漏隐含条件'的题拉出来看一遍"——这是真实的高效复习行为)。

**复练队列:遗忘曲线驱动(简化 SM-2)。**

- 间隔序列:错题产生后第 **1 / 3 / 7 / 15 / 30** 天各安排一次复练;
- **复练用变式不用原题**(数字重抽样版)——用原题会背答案,变式才检验真掌握;这就是 4.2 变式管线在产品里的主要出口;
- 复练通过(独立做对,不用提示)→ 进入下一间隔;不通过 → 间隔重置回第 1 天,连续两轮重置 → 升级为"回炉讲课"信号;
- 走完 30 天间隔 → 条目状态置 `mastered`,归档不再打扰;
- 每日复练队列封顶 5 题(超出的顺延),防止错题堆积把学生淹死——高三时间是最稀缺资源。

骨架版:间隔序列写死、变式即数字重抽样、两个视图做成两个简单列表页。完整版:间隔按个体表现自适应(答得快且对 → 间隔拉长)、考点卡片下钻到子错因、错题本与周计划模块联动(计划里自动排入"周三:斜面模型错题回炉")。

## 8. 诚实风险清单(当前技术明确做不到的事)

1. **AI 过程批改做不到 100% 准确。** 最优条件下与人一致性预计 85-93%;误判会发生,产品靠"低置信降措辞 + 申诉冻结 + 非常规解法豁免"消化,而不是假装不会错。
2. **AI 徒手画电路图/光路图不可用**,参数化库是唯一方案;参数化库覆盖不到的偏门图形,骨架版就是"贴截图或没有图"。
3. **手写识别约 88% 可接受率**,转写确认环节是必须的,会给学生增加一步操作,这是当前技术下的合理代价。
4. **AI 原创题无人审不可入库**,自动管线只能把通过率抬到 60-80%,"无限自动出题"在质量约束下不成立。
5. **错因归类到细子码的准确率(70-85%)只够做统计趋势**("你建模错偏多"可信),单次归因可能不准,呈现措辞避免把单次归因说成定论。
6. **难度定标初期纯靠人工经验**,数据校准需要几千次作答积累,冷启动阶段难度感受可能有偏,靠会话内动态规则(升降档)兜底。
