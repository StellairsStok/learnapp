# 高考物理 AI 私教(learnapp)技术架构设计方案 v1.0

> 写给两类读者:不写代码的创始人(每个决策都有"人话版"解释),和将要写代码的工程师(有数据结构、字段、流程和示例)。全文中【骨架版】= 第一个版本必须有的最小实现,【完整版】= 后续逐层叠加的目标形态。

---

## 0. 一句话回答创始人的问题

**"我 Claude 直接用你不可以吗?"——不可以,因为你平时聊天用的那个 Claude,只是这个产品的"发动机";而一辆车除了发动机,还需要底盘、方向盘、油箱和仪表盘。**

这个 app 的本质是:**Claude API(发动机)+ 四样外挂骨架**:

1. **权威知识清单**(课标考点树 + 知识卡 + 题库)——告诉它"教什么、边界在哪";
2. **学生记忆**(档案、掌握度、错题史)——让它"记得这个学生";
3. **教学流程**(诊断→讲→练→批→归因→调计划的状态机)——让它"按教学法办事"而不是有问必答;
4. **练习界面**(答题、拍照、公式渲染、进度页)——让学习行为能规范地发生和被记录。

这四样,恰恰是裸用 Claude 完全没有的。

## 1. 三个"Claude"的区别:你聊天用的那个,和产品要用的那个,不是一回事

### 1.1 三个东西,一个大脑

| 名称 | 它是什么 | 类比 | 适合谁用 |
|---|---|---|---|
| **claude.ai 聊天网页** | Anthropic 做好的一个"聊天产品",里面装着 Claude 模型 | 你去咨询台找一位很博学的人**随便聊** | 普通用户 |
| **Claude Code** | Anthropic 做的一个"编程助手产品",也装着 Claude 模型,但被设计成会读写文件、跑命令、写代码 | 同一个人,被安排到**工程师岗位**,配了一套工具 | 开发者(也就是现在跟你协作的我) |
| **Claude API** | 不带任何界面的"裸模型接口"。程序发一段文字/图片过去,它返回一段文字。按用量(token)计费 | 同一个大脑,装进**你自己造的机器里**,你决定它看到什么、说什么、什么时候说 | 做产品的人 |

三者共享同一个大脑(模型),区别在于**外面包了什么壳**。claude.ai 的壳是 Anthropic 设计的通用聊天壳;Claude Code 的壳是编程壳;**你的 app,就是你自己设计的"高考物理私教壳"**,壳里面调用的是 Claude API。

### 1.2 "名校毕业生 vs 特级教师"

裸的 Claude 模型,像一个**物理学得极好的名校毕业生**:知识渊博、反应快、什么都能聊。但把他直接扔进教室,他会:

- 不认识任何一个学生,每节课都当第一次见面;
- 学生问什么答什么,学生不问就不管;
- 讲题时随手用大学的拉格朗日方法,因为"这样更优雅";
- 下课就把一切忘光,不布置作业、不追作业、不复盘。

**特级教师和毕业生的差别,从来不在知识量,而在于:知道教学大纲的边界、记得每个学生的底细、有一套固定的教学章法、并且会主动安排下一步。** 这四样都不是"大脑"自带的,是"职业训练+工作制度"给的。

**模型提供知识和语言能力;产品提供教学的行为方式和边界。** 我们要做的全部工程工作,一言以蔽之:把毕业生训练成特级教师的那套"工作制度",用代码和数据固化下来。

### 1.3 还有一个非常现实的原因:claude.ai 没法变成你的产品

即使不谈教学质量:claude.ai 是 Anthropic 的产品,你不能改它的界面、不能接你的题库、不能读写你的学生数据库、不能给学生发账号、更不能收费。要做"你的产品",唯一的路就是 Claude API + 自建骨架。

## 2. 裸用 Claude 的六个翻车现场,以及每个翻车对应要补的骨架

| # | 翻车现场(真实会发生) | 根因 | 对应骨架组件 |
|---|---|---|---|
| 1 | 学生周一说"我万有引力很差",周三再来,Claude 完全不记得,重新问一遍"你哪里薄弱?" | API 是**无状态**的:每次调用都是全新的,模型没有任何跨会话记忆 | **学生档案库**(§5):档案+掌握度+错题史持久化到数据库,每次对话开始时注入 |
| 2 | 每次开新对话都要重新贴题目、重新解释"你是物理老师"、重新说明学生情况,聊 50 轮后开始"失忆"前面的内容 | 上下文窗口有限,且没人替它管理 | **会话管理 + 上下文策略**(§5.4):历史裁剪、课后总结回写、prompt 缓存 |
| 3 | 讲"带电粒子在磁场中运动"时,顺手用了微积分推导,或用了课标已删除的知识点;学生在考场上用了拿不到过程分 | 模型的知识是"全人类物理",它**不知道 2025 修订版课标的边界**,更不知道你所在省的等级考要求 | **课标考点树白名单**(§4):每次讲课只喂给它当前考点的知识卡,并在系统提示里硬性约束"只可使用清单内方法" |
| 4 | 学生说"讲讲动量守恒",Claude 讲得很好——但它**永远不会主动说**"根据你上周的错题,今天应该先补'系统选取',然后做这 4 道题" | 聊天模型是**被动应答**的,没有目标、没有计划、没有"教学主权" | **教学流程状态机 + 计划引擎**(§6):产品决定"现在处于哪个环节、下一步做什么",模型只负责执行当前环节 |
| 5 | 学生把解题过程拍照发过去,Claude 批改:有时把对的判错(Khanmigo 早期和学而思都翻过这个车),有时凭感觉给"错因" | 模型做数值运算和自由批改会**幻觉**;没有标准答案参照、没有错因标签约束 | **批改管线**(§6.4):题库自带标准答案+评分点,批改时注入;错因只能从固定标签集里选;结构化输出+置信度+人工抽检 |
| 6 | 想让学生"做 5 道题、逐题作答、看正确率、错题进错题本"——在聊天框里根本没法规范地做这件事 | 聊天界面不是练习界面 | **前端练习界面**(§3):答题卡、拍照上传、KaTeX 公式渲染、计时、进度页、错题本 |

记住这张表:**产品要做的每一个组件,都是为了堵住裸模型的一个具体的洞。** 没有多余的工程。

## 3. 整体架构

### 3.1 给创始人的一张图

```
 学生(浏览器)
    │  ①看讲解/答题/拍照
    ▼
┌─────────────────────────── 你的网页应用 ───────────────────────────┐
│  前端(界面层)                                                      │
│   聊天+讲课界面 │ 练习答题卡 │ 拍照上传 │ 学习报告/错题本 │ 公式渲染   │
├────────────────────────────────────────────────────────────────────┤
│  后端(大脑的"工作制度")                                             │
│   教学流程状态机:现在该诊断?讲课?练习?批改?归因?调计划?          │
│   每次调用 AI 前,负责"备课":                                        │
│     从【知识库】取出今天要讲的考点卡片+题目+标准答案                    │
│     从【学生记忆】取出这个学生的档案摘要                               │
│     拼成一份完整"教案"发给 Claude API                                 │
│   拿到回复后,负责"记录":更新掌握度、错题本、学习计划                   │
├──────────────┬─────────────────────┬───────────────────────────────┤
│  知识库(文件) │   学生记忆(数据库)    │   模型适配层(可换发动机)        │
│  课标考点树    │   学生档案            │   Claude API(现在)            │
│  知识卡片      │   掌握度表            │   国产模型(将来合规需要时切换)  │
│  题库+标准答案 │   错题史/会话史        │                               │
└──────────────┴─────────────────────┴───────────────────────────────┘
```

**人话版:** Claude API 只是右下角那一小格。学生每说一句话,后端先"备课"(从知识库和学生档案里取材料拼给模型),模型回答后,后端再"记录"(把学到的东西写回数据库)。模型永远不直接碰数据库和题库——它只看到我们喂给它的那份"教案"。这就是"有记忆、有边界"的实现原理:**记忆在数据库里,边界在喂料规则里,不在模型里。**

### 3.2 给工程师的具体选型

| 层 | 选型 | 理由 |
|---|---|---|
| 前端+后端框架 | **Next.js 14+(App Router)+ TypeScript** | 单仓库、单进程搞定前后端;本机 `npm run dev` 即可自测;Claude Code 对该栈生成质量高 |
| UI | Tailwind CSS + shadcn/ui | 快速出可用界面 |
| 公式渲染 | **KaTeX**(react-katex) | 高中物理公式复杂度低,KaTeX 同步渲染、体积小、够用(调研摘要二结论);个别页面需要再降级 MathJax |
| 数据库 | **SQLite**(better-sqlite3 或 Drizzle ORM + SQLite) | 自测阶段零运维、单文件、好备份;schema 设计时保持与 Postgres 兼容 |
| 知识库存储 | **仓库内文件**:YAML(考点树)+ Markdown(知识卡)+ JSON(题库),git 版本管理 | 教研内容就是代码资产,可 diff、可回滚、可让创始人直接在文件里改 |
| AI SDK | **@anthropic-ai/sdk**(官方 TypeScript SDK) | 流式输出、结构化输出、prompt caching、多模态开箱即用 |
| 图片处理 | 前端压缩(browser-image-compression)→ base64 → Claude 多模态 | 骨架版不接 Mathpix(调研结论:Claude 直接识别手写已达可用水平) |
| 物理示意图 | 骨架版:**预制 SVG 图库**(知识卡里手工放图);完整版:LLM 输出 JSON 中间表示 → 确定性渲染器画受力图/电路图 | "一步生成 SVG"不可靠是已验证结论 |
| 部署(自测) | 本机 Node 进程,`localhost:3000` | 见 3.3 |

### 3.3 自测阶段:全部跑在本机的形态

```
你的电脑
├── learnapp/                    ← 一个 git 仓库
│   ├── app/                     ← Next.js 前后端代码
│   ├── content/                 ← 知识库(教研资产,重点保护)
│   │   ├── syllabus/kp-tree.yaml       考点树
│   │   ├── cards/KP-*.md               知识卡片
│   │   └── questions/*.json            题库
│   ├── data/learnapp.db         ← SQLite(学生记忆,不进 git)
│   └── .env.local               ← ANTHROPIC_API_KEY(绝不进 git)
└── 浏览器打开 http://localhost:3000 → 创始人扮演学生实测
```

- 唯一的外部依赖是调用 Claude API 的网络请求。其余一切(界面、数据库、知识库)都在本机。
- **成本控制:** 自测阶段一天几十次调用,月成本个位数美元。
- **备份:** `content/` 随 git;`data/learnapp.db` 每天复制一份即可。

### 3.4 上线升级路径(自测通过之后)

| 阶段 | 形态 | 要做的事 |
|---|---|---|
| A. 本机自测(现在) | localhost | 无 |
| B. 少量真实学生内测(5-20 人) | 部署到一台云服务器(或 Vercel 等托管);SQLite → **PostgreSQL**;加登录(手机号/账号密码)、HTTPS | 数据库迁移脚本;简单账号系统;每日备份 |
| C. 公开/收费 | 境内服务器 + 域名备案;模型切换到已备案国产模型(见 3.5);未成年人合规功能(见 §9) | ICP 备案、算法/生成式 AI 备案、监护人同意流程、支付 |

代码层面为此预留的两件事,从第一天就要做:① 数据访问全部走 ORM/仓储层(换数据库只改配置);② 模型调用全部走适配层(换模型只改配置)。除此之外**不要**为上线提前做任何事——骨架阶段的敌人是复杂度。

### 3.5 模型可替换层(将来切国产模型的关键设计)

**为什么必须有:** 两个原因。一,面向中国公众提供生成式 AI 服务,实践上需要使用已完成备案的模型,Claude API 不在其列——所以商业化时大概率要把"主讲发动机"换成国产模型(DeepSeek、Qwen、GLM 等);二,不同任务用不同档次模型省钱(§7),本来就需要一个统一的调用入口做路由。

**设计:** 定义一个与厂商无关的内部接口,所有业务代码只面向它:

```typescript
// lib/llm/types.ts —— 业务代码只 import 这里的东西
export type TaskKind = "tutor_chat" | "grade" | "diagnose_attribution"
                     | "extract_ocr" | "classify" | "summarize_session";

export interface LlmRequest {
  task: TaskKind;
  system: SystemBlock[];          // 分块,便于做缓存断点
  messages: ChatMessage[];        // 支持 text + image
  jsonSchema?: object;            // 需要结构化输出时传
  stream?: boolean;
}
export interface LlmProvider {
  chat(req: LlmRequest): Promise<LlmResponse>;        // 或 AsyncIterable 流式
}
```

```typescript
// config/model-routing.ts —— 换模型 = 改这张表,业务代码零改动
export const ROUTING: Record<TaskKind, { provider: string; model: string }> = {
  tutor_chat:            { provider: "anthropic", model: "claude-sonnet-5" },
  grade:                 { provider: "anthropic", model: "claude-sonnet-5" },
  diagnose_attribution:  { provider: "anthropic", model: "claude-opus-4-8" },
  extract_ocr:           { provider: "anthropic", model: "claude-haiku-4-5" },
  classify:              { provider: "anthropic", model: "claude-haiku-4-5" },
  summarize_session:     { provider: "anthropic", model: "claude-haiku-4-5" },
};
```

适配层实现两个 Provider:`AnthropicProvider`(现在,官方 SDK)和将来的 `OpenAICompatProvider`(国产模型多数提供 OpenAI 兼容接口)。**关键纪律:任何业务代码不得直接 import Anthropic SDK。** 结构化输出、prompt 缓存这些厂商特性,都封装在 Provider 内部(国产模型不支持某特性时在适配层降级,如用"请只输出 JSON"提示词 + 解析重试来模拟结构化输出)。

**诚实提示:** 切换国产模型不是"改个配置就完事"——提示词需要针对新模型重新调优,教学质量需要重新走一遍你的测试用例。适配层保证的是"切换成本是几天,而不是重写"。

## 4. 知识库设计:考点树、知识卡、题库

这是产品的**护城河资产**(调研摘要二版权结论:题目可能是别人的,但考点体系、解析、错因标注是你自己的作品)。权威锚点 = 《普通高中物理课程标准(2017 年版 2025 年修订)》的"内容要求"+"学业质量"(调研摘要一结论),不锚定任何一版教材。

### 4.1 考点树(`content/syllabus/kp-tree.yaml`)

```yaml
- id: KP-X1-01-03            # 选择性必修1-主题01-第3个考点
  name: 动量守恒定律及其应用
  module: 选择性必修1
  theme: 动量与动量守恒
  curriculum_text: "……(课标'内容要求'条目原文,创始人从官方PDF逐条录入)"
  exam_scope: 等级考           # 合格考 | 等级考(依据学业质量分级,2025修订版待终核)
  prerequisites: [KP-B1-02-04, KP-X1-01-01]
  methods_allowed: ["动量定理", "动量守恒(一维/二维分解)", "碰撞分类讨论"]
  methods_forbidden: ["微积分推导", "质心系解法"]   # 超纲方法黑名单,讲课时注入
  textbook_map: { 人教: "选必1 第1章第3节", 粤教: "…" }   # 完整版再补全5版
```

- 【骨架版】只建**一个板块**(建议力学中的"动量"或"曲线运动与万有引力",约 15-25 个考点节点),把闭环跑通;【完整版】覆盖全部必修+选必约 100+ 条内容要求。
- `methods_allowed/forbidden` 是"不超纲"的核心机制:讲课和批改的系统提示里逐条注入,并要求模型自检(§8)。

### 4.2 知识卡片(`content/cards/KP-X1-01-03.md`)

一个考点一张卡,就是"这节课的标准教案",讲课时整卡注入模型:

```markdown
---
kp_id: KP-X1-01-03
version: 3
reviewed_by: founder        # 创始人审核标记,未审核的卡不允许上线
---
## 核心结论(考试可直接使用的)
- 系统不受外力或所受外力矢量和为零时,系统总动量守恒:p₁+p₂=p₁'+p₂' …
## 适用条件与常见误区
- 误区1:忽略"系统"的选取,把内力当外力 …
## 标准解题流程
1. 选系统 → 2. 判条件(守恒/某方向守恒)→ 3. 规定正方向 → 4. 列方程 …
## 教学脚本提示(给AI的教法指令)
- 先用"冰面上推人"生活情境引入;讲完条件判断后必须让学生口头复述一遍再继续
## 典型例题
[Q-0142, Q-0143]
## 预制示意图
![碰撞前后示意](../assets/kp-x1-01-03-fig1.svg)
```

### 4.3 题库(`content/questions/*.json`)

```json
{
  "id": "Q-0142",
  "kp_ids": ["KP-X1-01-03"],
  "difficulty": 3,
  "type": "计算题",
  "source": { "kind": "adapted", "note": "改编自2023某省题,已改数值与情境" },
  "province_tags": ["通用"],
  "stem_md": "质量为 $m_1=2\\,\\text{kg}$ 的滑块……",
  "figure": "assets/q-0142.svg",
  "standard_answer_md": "……(完整规范解答)",
  "grading_rubric": [
    { "point": "正确选取系统并判断动量守恒条件", "score": 3 },
    { "point": "规定正方向并正确列出守恒方程", "score": 4 },
    { "point": "代入求解,数值与单位正确", "score": 3 }
  ],
  "common_errors": ["未判断守恒条件", "方向符号错误", "计算失误"],
  "numeric_answer": { "value": 1.5, "unit": "m/s", "tolerance": 0.01 }
}
```

**每道题必须有 `standard_answer_md` + `grading_rubric` + `numeric_answer`,没有就不许进批改管线**——这是 Khanmigo 用真金白银换来的教训(批改前注入人工标准答案,准确率显著提升)。`numeric_answer` 供后端**用代码直接比对数值**,不依赖模型口算。

- 【骨架版】100-200 题,自编+改编(版权安全区),覆盖所选板块;检索方式就是按 `kp_ids`+`difficulty` 数据库查询,**不需要向量检索**。
- 【完整版】扩到 2000+ 题、按省份/卷种打标签、加向量检索做"相似题推荐"。

## 5. 学生记忆的实现

### 5.1 原则:记忆在数据库,不在对话里

模型没有记忆,所以"记忆"= **数据库里的结构化事实** + **每次调用前生成的一段"学生摘要"注入到提示词里**。原始聊天记录不是记忆,是原材料;课后要蒸馏成结构化数据。

### 5.2 数据结构(SQLite,字段即文档)

```sql
CREATE TABLE students (
  id TEXT PRIMARY KEY, name TEXT, province TEXT,        -- 决定卷种/赋分模式
  exam_mode TEXT,          -- '3+1+2' | '3+3'
  target_score INTEGER, textbook TEXT,                  -- 人教/粤教/…
  exam_date TEXT,          -- 高考日期,计划引擎的倒计时锚点
  profile_notes TEXT, created_at TEXT
);

CREATE TABLE mastery (                                  -- 掌握度:记忆的核心
  student_id TEXT, kp_id TEXT,
  score REAL DEFAULT 0.3,      -- 0~1,启发式估计(见5.3),不是真理
  attempts INTEGER DEFAULT 0, correct INTEGER DEFAULT 0,
  last_practiced_at TEXT, updated_at TEXT,
  PRIMARY KEY (student_id, kp_id)
);

CREATE TABLE attempts (                                 -- 每一次作答的原子记录
  id TEXT PRIMARY KEY, student_id TEXT, question_id TEXT, session_id TEXT,
  answer_text TEXT, answer_image_path TEXT,
  verdict TEXT,            -- correct | partial | wrong | skipped
  score REAL, error_types TEXT,   -- JSON数组,值只能来自错因标签集
  ai_feedback TEXT,        -- 批改JSON全文,供复查/申诉
  needs_review INTEGER DEFAULT 0, created_at TEXT
);

CREATE TABLE sessions (
  id TEXT PRIMARY KEY, student_id TEXT,
  kind TEXT,               -- diagnose | lecture | practice | review
  kp_ids TEXT, started_at TEXT, ended_at TEXT,
  summary TEXT             -- 课后AI蒸馏的总结(≤300字),下次课的记忆来源
);

CREATE TABLE messages (    -- 会话内原始对话,仅用于会话内上下文与事后审计
  id INTEGER PRIMARY KEY, session_id TEXT, role TEXT,
  content TEXT, created_at TEXT
);

CREATE TABLE study_plan (
  student_id TEXT, plan_date TEXT,
  items TEXT,              -- JSON: [{kind:'review',kp_id:…},{kind:'practice',question_ids:[…]}]
  status TEXT, generated_by TEXT,   -- 记录由哪次归因触发生成
  PRIMARY KEY (student_id, plan_date)
);
```

错因标签集(【骨架版】固定 8 个,写死在代码里;完整版参照讯飞思路逐步细化到"考点×错因"):
`concept_gap 概念不清 | formula_misuse 公式误用 | modeling_error 建模/过程分析错误 | calculation_slip 计算失误 | misread 审题偏差 | unit_sign 单位与符号错误 | incomplete 未完成/未作答 | out_of_scope_method 使用超纲方法`

### 5.3 掌握度更新(骨架版:简单可解释,拒绝伪精确)

```
做对:score ← score + 0.25 × (1 - score) × difficulty_weight
做错:score ← score - 0.30 × score
部分对:按得分率线性插值
时间衰减:读取时 score_eff = score × exp(-0.01 × 距上次练习天数)
```

**给创始人的诚实说明:** 这不是心理测量学意义上的能力估计(IRT/知识追踪是完整版的事),它只是"够用的排序器"——能把薄弱考点排到前面就完成任务了。前 3 个月不值得为更精确的模型投入。

### 5.4 每次对话如何注入记忆:三段式提示词 + 缓存

每次调用模型,后端按固定顺序拼装(顺序即缓存友好度,稳定在前、易变在后):

```
[system 第1块|全局静态,标注 cache_control] 教师人设 + 教学法总纲 + 超纲红线 + 输出规范
[system 第2块|会话内静态,标注 cache_control] 本节课知识卡全文 + 相关题目与标准答案 + methods_allowed/forbidden
[system 第3块|会话内静态] 学生档案摘要(下例,由后端从数据库生成,600-1200 token)
[messages] 本会话对话历史(易变部分)
```

学生档案摘要示例(程序模板生成,不靠模型现编):

```
【学生档案】小明|广东|3+1+2(物理原始分计入)|目标80+|距高考340天|人教版
【掌握度】薄弱TOP5:动量守恒0.31、动生电动势0.38、天体运动0.42、…(练过的考点共23个)
【近7天】练32题,正确率56%;高频错因:建模错误×9、计算失误×6
【上次课摘要】讲动量定理;学生在"过程选取"处两次卡壳,已布置Q-0144未完成
【今日计划】复习KP-X1-01-03知识卡 → 练Q-0151(易)Q-0142(中)Q-0155(难)
```

### 5.5 上下文长度管理策略

| 策略 | 规则 |
|---|---|
| 会话即课时 | 一次"课"= 一个 session,自然隔离上下文;下课即总结 |
| 会话内裁剪 | 对话超过 ~30 轮或估算 40K token:保留系统块+最近 12 轮,更早的由后端用 Haiku 压成 300 字"前情提要"插入 |
| 课后蒸馏 | session 结束(显式下课或闲置 30 分钟)触发后台任务:Haiku 生成 `sessions.summary`(学了什么/卡在哪/待办),同时结算本课 attempts → 更新 mastery |
| 跨会话记忆 | 永远只通过"档案摘要"进入新对话,**绝不**把历史聊天记录原文拼进新会话 |
| 缓存 | 系统块 1、2 打 `cache_control` 断点;同一节课内多轮对话,知识卡等大块内容按缓存读价(约 0.1×)计费,这是成本控制的主要手段 |

## 6. 核心流程逐个拆解(闭环六环节)

产品的"教学主权"由后端状态机掌握:`diagnose → plan → lecture → practice → grade → attribute → (更新 plan) → …`。模型永远只被要求执行**当前环节的单一任务**。

### 6.1 摸底诊断

- 【骨架版】固定诊断卷:每个考点 2-3 题、由易到难,程序按题库标签抽题;学生作答→走批改管线→初始化 mastery 表。不搞自适应。
- 【完整版】自适应诊断(答对跳更难/答错下探先修考点 prerequisites),配合"学生自评+目标访谈"对话。

### 6.2 计划

- 【骨架版】规则引擎(纯代码,不用模型):每日计划 = 按 `score_eff` 升序取薄弱考点 1-2 个 + 遗忘曲线到期复习项(1/3/7/14 天间隔)+ 每考点按难度阶梯抽 4-6 题。可解释、可预测、不会幻觉。
- 【完整版】规则引擎产出草案 → Opus 审阅微调并生成"给学生的一段计划说明"(模型只做解释和微调,不拥有决定权)。

### 6.3 讲课

- 流式输出(打字机效果);系统提示要求苏格拉底式:讲一段→提一个检查性问题→根据回答决定继续或回炉;**不许直接报答案**,学生索要答案时先给分层提示(Khanmigo 模式)。
- 超纲红线注入:"你只能使用【methods_allowed】中的方法;禁止使用【methods_forbidden】及任何微积分。"
- 【骨架版】示意图用知识卡预制 SVG;【完整版】接"JSON 中间表示→确定性渲染器"的受力图/电路图管线。

### 6.4 批改(可靠性设计重点,§8 有失败模式)

管线(每一步都是独立、可测试的函数):

1. **输入**:学生文字作答,或照片(前端压缩后 base64)。
2. **OCR 提取**(Haiku,多模态):照片→转写为文字+LaTeX,`transcription_confidence` 低于阈值时,**先把转写结果显示给学生确认**("我看到你写的是……对吗?")再进入批改——把 OCR 错误在源头拦住。
3. **数值预判**(纯代码):从转写中抽取最终数值答案,与 `numeric_answer` 容差比对。**算术对错不问模型。**
4. **结构化批改**(Sonnet + structured outputs):注入题目、标准答案、rubric、错因标签集、学生答案,要求按 schema 输出:

```json
{
  "verdict": "partial",
  "score": 6, "full_score": 10,
  "rubric_results": [
    {"point": "正确选取系统并判断守恒条件", "earned": 3, "comment": "判断正确"},
    {"point": "规定正方向并列方程", "earned": 3, "comment": "方向未声明,方程符号混乱,扣1分"},
    {"point": "数值与单位", "earned": 0, "comment": "计算错误(数值比对已由系统确认)"}
  ],
  "error_types": ["unit_sign", "calculation_slip"],
  "feedback_to_student": "……(鼓励+指出关键一步)",
  "confidence": 0.82
}
```

5. **一致性校验**(纯代码):rubric 得分和≠score、error_types 含标签集外的值、模型判"数值正确"但第 3 步判错——任一触发则重试一次,仍不一致则 `needs_review=1`,前端显示"该题批改待复核"。
6. **落库**:写 attempts,更新 mastery。

### 6.5 错题归因 + 6.6 调整计划

- 逐题错因已由批改产出(标签约束)。**每日归因**是单独一次 Opus 调用:输入近 7 天 attempts 汇总 + mastery 快照,输出结构化的"根因假设"(例:"动量方向处理连错 3 次,根因更可能是矢量运算基础(先修 KP-B1-02-01)而非动量概念本身")+ 对计划引擎的调整建议(结构化字段:`insert_review_kp`, `demote_kp`, `difficulty_adjust`)。
- 计划引擎消费这些建议后重新生成次日 study_plan;归因的自然语言解释展示给学生("为什么今天安排你复习矢量分解")——**这正是"私教感"的来源,也是线下老师最值钱的动作。**

## 7. 模型分工与成本估算

### 7.1 各任务用什么档次

> 单价来自本机开发工具内置的价格表(**缓存日期 2026-06-24**,美元/百万 token)。**具体单价与最新优惠请在主会话对照官方 pricing 页核实后再做商业测算。**

| 模型 | 输入 | 输出 | 定位 |
|---|---|---|---|
| Claude Opus 4.8 | $5 | $25 | 最强推理 |
| Claude Sonnet 5 | $3(2026-08-31 前优惠 $2) | $15(优惠 $10) | 质量/成本平衡,接近上代 Opus 水平 |
| Claude Haiku 4.5 | $1 | $5 | 快、便宜 |

| 任务 | 模型 | 理由 |
|---|---|---|
| 讲课/苏格拉底对话(调用量最大) | **Sonnet 5** | 教学对话质量要求高但知识卡已注入,Sonnet 足够;量大,用 Opus 成本×~1.7 |
| 批改 | **Sonnet 5** | 有标准答案+rubric 兜底后,难度已从"解题"降为"对照",Sonnet 胜任;抽检发现质量不足再对疑难题型升级 Opus |
| 每日错题归因 + 计划调整 | **Opus 4.8** | 每生每天 1 次、输入长、推理深,是"私教含金量"所在,值得用最好的 |
| 拍照 OCR 转写 | **Haiku 4.5** | 转写任务,便宜且快;置信度低时该题升级 Sonnet 重转 |
| 课后总结、打标签、分类 | **Haiku 4.5** | 机械任务 |
| 夜间批量任务(周报、全量计划重算) | 同上模型 + **Batch API** | 批量接口五折,不赶时间的都走它 |

自测阶段的省心做法:先全用 Sonnet 5 跑通,再按上表分层——不要一开始就优化成本。

### 7.2 成本估算框架(每生每天学习 1 小时)

**假设**(创始人可自行代入调整;汇率按 1 USD ≈ ¥7.2):

| 活动 | 次数/天 | 单次输入(未缓存口径) | 单次输出 | 模型 |
|---|---|---|---|---|
| 讲课对话 | 20 轮 | ~6K(系统块+知识卡+摘要+历史) | ~400 | Sonnet |
| 批改(含图) | 6 题 | ~3.5K(题+标答+rubric+作答图) | ~800 | Sonnet |
| OCR 转写 | 6 次 | ~1.8K(图) | ~300 | Haiku |
| 归因+计划 | 1 次 | ~8K | ~1.5K | Opus |
| 课后总结 | 1 次 | ~15K(全对话) | ~0.4K | Haiku |

**未用缓存的日成本**:Sonnet 输入 ~141K、输出 ~13K → $0.42+$0.19;Opus → $0.04+$0.04;Haiku → $0.03。合计 **≈ $0.72/天 ≈ ¥5.2/天**。

**启用 prompt 缓存后**(讲课 20 轮里,知识卡+人设块 19 次按 0.1× 计):输入成本约降 60-70% → **≈ $0.35-0.45/天 ≈ ¥2.5-3.3/天**。

**月成本区间(30 天、每天 1 小时):**

| 方案 | 月成本/生 |
|---|---|
| 全 Sonnet + 缓存 | **≈ ¥75-100** |
| 分层(上表)+ 缓存 | **≈ ¥70-110** |
| 全 Opus(上限锚点) | ≈ ¥170-250 |
| 数量级结论 | **每月每生 ¥50-250 之间,即每小时 ¥2-8** |

**对比线下:** 一对一物理补课 ¥200-500/小时 × 30 小时/月 = **¥6,000-15,000/月**。AI 成本约为其 **1%-2%**。即使估算偏差 3 倍、即使学生每天学 2 小时,结论不变:**成本结构上,这个产品成立;真正的风险从来不在 API 账单,而在教学质量(§8)。**

## 8. 诚实清单:AI 做不到什么、会怎么失败、我们怎么补救

这一节是给创始人的"防幻灭疫苗"。以下每一条都是行业已付过学费的真实短板。

| # | 做不到"完美"的地方 | 真实水平 | 失败模式 | 补救机制(已设计进架构) |
|---|---|---|---|---|
| 1 | **批改绝对可靠** | 有标准答案+rubric 约束下,预期 90-97% 可接受(讯飞公开数据 97% 是其多年错因库加持的上限,我们初期达不到) | 把对判错、把错判对、扣分点张冠李戴(学而思、Khanmigo 均翻过车) | 数值比对走代码不走模型;rubric 逐点批改;一致性校验+置信度;`needs_review` 队列;前端"对批改有异议?"按钮;创始人每周抽检 20 份 |
| 2 | **数值计算** | 模型口算多步运算会出错,概率不可忽略 | 讲题时算错数,权威感崩塌 | 凡是判定对错的数值,一律代码比对;讲解中的示例计算在题库标答中预先写死;完整版可加代码执行工具 |
| 3 | **手写 OCR** | 工整手写 ~90% 可接受转写;潦草、多次涂改会明显下降 | 转写错→批改跟着全错,学生被冤枉 | 低置信度先让学生确认转写再批改;引导"最终答案写清楚、拍正" |
| 4 | **绝对不超纲** | 白名单+黑名单约束后大幅收敛,但无法保证 100%——模型偶尔仍会溜出一个课标外结论 | 讲了"二级结论速成"或课标已删内容 | 每张知识卡人工审核(创始人把关);讲课系统提示强约束;前端"这个方法我们没学过"举报按钮→进人工复核;完整版加"生成后超纲检查器"(Haiku 对输出做二次扫描) |
| 5 | **物理示意图生成** | 一步生成 SVG 不稳定(行业共识) | 受力箭头画错方向、电路图错乱 | 骨架版只用人工预制图;完整版走 JSON 中间表示+确定性渲染器 |
| 6 | **掌握度的"科学性"** | 骨架版是启发式打分,不是测量 | 个别考点被高估/低估,计划偶尔不精准 | 用于排序而非绝对判断;学生可手动标记"我觉得我会了/不会";数据攒够后再上正经知识追踪模型 |
| 7 | **教学风格稳定性** | 提示词能约束大方向,但语气、详略会有波动 | 有时啰嗦、有时跳步 | 教学法提示词版本化管理+创始人持续实测调优;固化"讲-问-答"节拍到流程而非全靠提示词 |
| 8 | **"感知学生情绪/状态"** | 只能从文字推断,会误判 | 学生已经很挫败,AI 还在追问 | 界面提供显式出口:"太难了,换个讲法/今天到这吧"按钮,把状态表达变成点击而不靠模型猜 |

**总原则:凡是能用确定性代码做的,绝不交给模型;凡是交给模型的,都要有 schema 约束、置信度、人工抽检和用户申诉四道后手。**

## 10. 骨架版 vs 完整版总表

| 子系统 | 骨架版(第一版就要有) | 完整版(叠加方向) | 升级路径 |
|---|---|---|---|

## 12. 结语:重述给创始人

- 你聊天时感受到的"Claude 很强",是真的——但那是**发动机的强**。裸发动机不记得学生、不懂课标边界、不会主动安排复习、也没有练习的场地。
- 我们要做的 app = **Claude API + 权威知识清单 + 学生记忆 + 教学流程 + 练习界面**。每一个组件都对应裸模型的一个具体缺口,没有一件是"为了做 app 而做 app"。
- 成本上完全成立:每生每天 1 小时,AI 成本约 **¥2-8/小时**(单价待主会话核实),是线下一对一的 1%-2%。
- 最大的风险不是技术跑不通,而是**批改可靠性和教学手感**——所以架构里到处是"代码兜底、标答注入、置信度、人工抽检、学生申诉"这样的保险丝,以及需要你长期投入的教研审核环节。
- 架构为两件未来的事预留了口子:**换国产模型**(合规)和**上云多用户**(增长)。除此之外,第一版一切从简:一个板块、一百多道题、一个能跑完"诊断→讲→练→批→归因→调计划"闭环的本机网页。先让你自己,被它教会一章物理。
